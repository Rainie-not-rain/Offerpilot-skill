# offerpilot_skill 提示词｜AI Coding 岗位

> 适用于 AI Coding 应用开发、AI 原型、Vibe Coding、技术产品原型和需要“产品 + 可运行 Demo”的岗位。复制提示词后补充方括号内容。

```text
请使用 `$offerpilot-skill`，针对以下 AI Coding / AI 应用原型岗位优化我的求职材料。

【目标公司与岗位】
[公司、岗位、团队/产品方向]

【岗位更偏向】
[产品原型 / 应用开发 / 工作流自动化 / 技术产品 / 尚不确定]

【目标市场与语言】
[国家/地区、中文或英文]

【JD】
[粘贴完整 JD；没有时写“暂无”，并先检索官方或同岗信息]

【我的简历、项目或代码材料】
[简历、项目说明、代码仓库、Demo、截图、PRD或文件路径]

【当前状态】
[尚未投递 / 已投不可修改 / 已收到面试；若有面试，仅填写已确认轮次]

请按以下流程执行：

1. 先做输入体检。识别项目是否真实可运行、我能否展示代码/Demo、目标用户、使用场景、技术栈、AI 工具、测试记录和个人贡献。缺口可降级时继续；需要深挖时每轮只问 2–4 个问题。
2. 从 JD 收敛 3–5 项核心能力，重点判断：真实问题定义、工作流拆解、AI Coding 使用、数据/状态设计、可运行交付、调试测试、产品取舍和工程边界。不要仅统计工具关键词。
3. 建立 JD→经历、Claim→evidence 和项目→可展示产物映射。证据可以是仓库提交、运行截图、Demo 链接、字段/接口说明、测试记录、Issue 或用户反馈；核对哪些代码由我编写、AI 生成、他人协作或模板复用。
4. 深挖项目时按“真实痛点→用户工作流→最小可用范围→页面/模块→数据实体与状态→AI Coding 过程→调试与验收→异常/权限/隐私→迭代”展开。要求我能解释设计理由，不要求假装底层工程专家。
5. 输出定向简历正文，并遵循“公司/项目｜岗位/角色｜时间 → 项目背景 → 2–3 个工作方向事项”的结构。项目背景用 1 句话交代用户、低效工作流、目标和 MVP 定位；每件采用“工作方向 + 业务价值”小标题，在同一段内用隐式 STAR 交代拆解、实现、调试/验收和结果。优先呈现：
   - 场景与范围：为谁解决什么人工或低效流程，为什么不做更大的系统。
   - 方案与实现：如何拆页面、模块、字段、状态、接口或自动化步骤。
   - 验证与迭代：如何调试、测试、处理错误提示、边界和用户反馈。
6. 不把“用了 AI Coding”直接等同于技术能力。准确说明我负责的需求拆解、Prompt、代码审阅、调试、集成、验收和产品判断；不会的架构或底层原理进入边界说明。
7. 无法证明上线、用户规模、效率提升或独立开发时，删除或降级，不用“成功落地”“大幅提升”等词替代证据。模板项目必须说明改动范围和原创贡献。
8. 自检固定事实、项目可运行性、代码归属、字段与流程一致性、测试证据、安全/隐私边界、JD匹配和可追问性。指出最值得补齐的 Demo、README、架构图、测试记录或录屏。
9. 若我需要面试准备，为每条最终 bullet 准备讲法和追问。重点预警：代码是不是你写的、AI 生成错误怎么处理、前后端/接口如何连接、字段为什么这样设计、如何验收、异常和权限怎么办、为什么不用现成工具。
10. 模拟面试一次只问一个问题，可要求我现场解释一个页面、字段或 bug。根据回答输出“事实是否站得住、技术边界、产品判断、表达优化、需补材料”。
11. 我补充代码、Demo 或面试反馈后，只更新受影响的 bullet、技术边界和题库，保持项目事实、简历和面试讲法一致。

最终先给可直接使用内容，再给证据状态、风险和补强动作；不要编造代码、仓库、用户、技术栈、性能指标或面试轮次。
```

## 岗位特异检查

- 项目是否来自真实场景，而非只为展示技术栈？
- 是否能演示一条端到端工作流和最小数据闭环？
- 是否能解释 AI 生成代码的审阅、调试和验收过程？
- 是否准确区分独立完成、AI 辅助、模板复用和团队协作？
- 是否准备了异常、权限、隐私和下一步扩展边界？
- 是否严格使用“项目背景 + 2–3 个工作方向事项”，而不是技术栈或功能清单？
