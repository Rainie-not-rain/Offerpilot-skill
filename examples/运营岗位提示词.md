# offerpilot_skill 提示词｜运营岗位

> 适用于内容运营、用户运营、活动运营、社区运营和增长运营。复制提示词后补充方括号内容。

```text
请使用 `$offerpilot-skill`，针对以下运营岗位优化我的求职材料。

【目标公司与岗位】
[公司、岗位、业务/产品方向]

【运营类型】
[内容 / 用户 / 活动 / 社区 / 增长 / 尚不确定]

【目标市场与语言】
[国家/地区、中文或英文]

【JD】
[粘贴完整 JD；没有时写“暂无”，并先检索官方或同岗信息]

【我的简历/经历】
[粘贴简历，或提供文件路径]

【当前状态】
[尚未投递 / 已投不可修改 / 已收到面试；若有面试，仅填写已确认轮次]

请按以下流程执行：

1. 先做输入体检。优先抽取我已有的账号、社群、活动、内容、用户研究、数据分析、校园组织、兼职和个人项目。若需要深挖，每轮只问 2–4 个问题，并复述已确认事实。
2. 从 JD 收敛 3–5 项核心能力。根据岗位类型判断内容选题、用户分层、活动推进、社群机制、漏斗转化、数据诊断、复盘迭代和跨团队协作的优先级。
3. 建立 JD→经历和 Claim→evidence 映射。核对账号后台、统计时间、指标定义、样本范围、基线、外部影响和个人贡献；没有证据时不写涨粉率、转化率或增长归因。
4. 深挖每段经历时围绕“目标用户与业务目标→策略假设→具体动作→关键指标/定性证据→问题诊断→下一轮调整”提问。把流水账转成运营闭环，但不得把参与或协助机械改成主导。
5. 输出定向简历正文，并遵循“公司/项目｜岗位/角色｜时间 → 项目背景 → 2–3 个工作方向事项”的结构。项目背景用 1 句话交代用户、业务问题、目标和运营定位；每件采用“工作方向 + 业务价值”小标题，在同一段内用隐式 STAR 交代策略、动作、指标/证据与复盘。优先使用以下方向：
   - 内容：用户需求/选题策略→生产与分发→打开、互动或转化→复盘。
   - 用户/社群：分层与需求→触达/机制→活跃、留存或反馈→迭代。
   - 活动：目标与人群→传播/报名/到场链路→漏斗卡点→优化。
   - 增长：目标指标→实验或策略→结果与归因边界→下一步。
6. 无法确认的结果进入“需确认/不可使用”，不进入可投递正文。没有精确数字时可以使用可验证的规模、交付物或定性变化，但不得写“明显提升”来暗示不存在的结果。
7. 自检固定事实、岗位相关度、owner 边界、指标口径、归因、运营闭环、内容密度和可追问性。指出最值得补充的后台截图、活动记录、看板、内容案例或复盘材料。
8. 若我需要面试准备，为每条最终 bullet 准备 30 秒讲法、至少 2 个追问、二三层追问、证据、禁区和诚实兜底。重点预警：你具体负责什么、指标怎么算、为什么这么做、数据下降如何排查、结果能否归因、复盘后改了什么。
9. 模拟面试一次只问一个问题。根据我的回答追问目标、动作、数据和复盘；反馈时给出更自然的口语版本，但不要替我编造细节。
10. 我补充后台数据或面试反馈后，只更新受影响的 bullet、指标口径、回答和风险地图，并记录哪些内容仍未确认。

最终先输出可直接使用的简历内容，再输出证据风险、自检和下一步；不要因“文科/校园经历”而降低标准，也不要把校园规模包装成互联网业务规模。不要编造用户、数据、后台指标、owner 范围或运营结果。
```

## 岗位特异检查

- 是否明确目标用户、业务目标和运营类型？
- 是否至少呈现一次“数据/反馈→诊断→调整”的闭环？
- owner 是否限定到真实负责的链路，而非整个项目？
- 指标是否有定义、时间范围、基线与归因边界？
- 内容、活动和用户动作是否说明“为什么这样做”？
- 是否严格使用“项目背景 + 2–3 个工作方向事项”，且背景没有重复事项内容？
