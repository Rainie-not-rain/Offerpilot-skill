# offerpilot_skill 质量闸门

## 使用规则

最终交付前在内部执行。未通过时修正、降级或说明缺口，不把未完成稿称为终稿。质量闸门不能证明用户素材真实，只能追问、检查一致性和控制表达风险。

## P0红线

任一失败都必须停止最终交付：

| 红线 | 失败表现 |
| --- | --- |
| 造假 | 新增不存在的背景、经历、数据、结果、招聘流程或面试轮次 |
| 硬信息被改 | 擅自改变学校、学历、公司、职位、日期、身份 |
| 来源冒充事实 | 弱来源、旧信息或推断被写成确定事实 |
| 过度包装 | 支持工作写成owner；AI辅助写成独立实现；团队结果写成个人结果 |
| 数字失控 | 无口径数字、百分比/百分点混用、占位符进入可使用正文 |
| 轮次臆测 | 没有邀请或高置信来源却生成确定轮次准备 |
| Bullet漏项 | 用户要求完整面试准备时，实际投递简历存在未覆盖bullet |
| 内部标签泄漏 | 正文出现待确认标记、迁移标签、模板指令、旧版说明或工作记录 |
| 简历与面试矛盾 | 两边的角色、数据、方法、时间或结果不一致 |

## P1核心质量

### 岗位调研

- 已识别市场、语言和时间范围；不确定项有置信度。
- 核心能力收敛到3-5项，并有JD/来源依据。
- 关键动态使用近期一手或可靠来源，不使用裸记忆。
- 来源覆盖说明包含等级、时间、缺口和低置信结论。
- 面试轮次标注来源和置信度。

### 简历诊断与改写

- 用户通过对话框粘贴文本时，已先完成纯文本完整性检查，并识别基础信息、教育、实习/工作、项目、技能和自我评价，而非直接把局部经历当作完整简历。
- Markdown、CSV 或 xlsx 输入已按对应字段规则解析并重新执行完整性、事实和教育经历保真检查；CSV/xlsx 转换骨架未被直接当成终稿。
- 必填内容缺失时已优先从已确认材料可靠提炼；不能确认的内容已分轮询问，未用模型推断联系方式、实习档期、学校或项目事实。
- 用户明确没有实习/工作经历时允许省略该模块；项目经历缺失时已优先深挖真实课程、科研、比赛、校园或个人作品，没有用虚构项目补齐结构。
- 必要信息尚未补齐且用户未确认模块确实不存在时，没有生成或命名为最终版简历。

- 每个核心JD能力有直接、可迁移、弱或无证据判断。
- 每条新增/强化claim能追溯到用户事实。
- 每个数字有定义、样本/范围、时间、基线和归因；缺口已降级。
- 每个强动词有真实角色边界。
- 最相关经历获得最高排序和篇幅，低相关内容已压缩。
- 同一模块内的教育、实习/工作、项目及其他带日期经历已按结束时间倒序排列；仍在进行的经历优先，结束时间相同则按开始时间倒序。
- 输入内容超出简历篇幅时，已优先保留与目标JD最匹配且证据最强的经历；低相关、重复内容已压缩或删除，重要删减项已在解释区说明。
- 每段核心经历通常为2-3件高价值事项，并形成产品/业务逻辑与隐式STAR。
- 固定事实字段未被擅自改动。
- 可复制正文与解释、风险、缺失证据分离。
- 教育背景、项目经历、个人技能和自我评价齐全；实习/工作经历按实际有无决定是否保留。
- 用户提供且涉及学校的教育经历已逐段完整保留，不因一页目标删除、合并任何学校或学历阶段；DOCX 输入中的嵌套表格已递归解析并通过 XML/完整文本交叉检查。
- 用户提供的求职意向与实习可用时间位于基础信息下方，内容未被擅自补写。
- 每段实习/工作和项目都有一句项目背景；每条工作/项目事项和技能都有且仅有一个加粗小标题。工作/项目小标题只保留工作方向，不含“｜价值/结果”式后缀。
- 每条原始事项独立对应一条优化内容，通常保留2-3行信息密度，没有为版面过度删减关键动作、方法或结果。
- 经历数量超过一般一页预算时已向用户确认删减或保留，不静默丢弃。

### 面试准备

- 使用实际投递/最终简历建立bullet inventory。
- 每个bullet有30秒讲法、至少2个追问、1个二三层追问、证据口径、禁区和处理策略。
- 王牌/高风险项目有60-90秒展开、关键取舍和复盘。
- 自我介绍有口语版和关键词卡，不是简历朗读稿。
- 风险地图覆盖数字、职责、技术边界和结果归因。
- 独立轮次文件只对应已确认或高置信轮次。
- 面后复盘能把新问题写回后续准备。

## P2体验质量

- 结论在前，结构能快速扫读，长段落克制。
- 语言自然，不油腻、不羞辱、不虚假鼓励、不像AI模板。
- 文件名和标题面向用户，不暴露内部实现。
- 输出语言、格式和案例符合目标市场。
- 一页简历的视觉层级、留白和字号可读；生成文档已完成渲染检查。
- DOCX全文为黑色；中文为微软雅黑，英文/数字为 Times New Roman；模块分隔为1磅黑色下边框且无色块底纹。
- 用户提供证件照时使用真实图片；未提供时才使用替代块。照片或替代块高度2.6cm，固定在右上角，与姓名和基础信息同栏。
- 姓名、联系方式、求职意向和实习时间在信息区视觉居中，照片不会挤偏其中心位置。
- 所有表格均按当前页面可用窗口宽度自适应至100%，列宽按比例调整；不存在固定总宽导致的越界、异常换行、右侧留白或照片挤压。
- 模板与岗位侧重符合场景：校招/社招和 AI Agent、后端、算法、前端、产品、运营等方向的排序有真实 JD 或岗位预设依据。
- 使用 PDF 模板时仅选择 compact/classic/modern/timeline；未指定颜色时沿用案例默认色，指定预设色或自定义色值时正确覆盖大标题、分隔线、时间轴、项目符号或强调底色。
- PDF 渲染不依赖 CDN、远程字体、在线图标或在线简历网站；正文仍是可搜索、可选择、可复制的文本。
- 身份证号、银行卡号、详细家庭住址等非必要敏感字段未进入正文、front matter 或公开交付物。
- 经历/项目双列标题表格按当前 section 的页面可用宽度铺满正文窗口，左缩进为0，日期列宽度与右对齐一致；未使用固定页面宽度造成右侧留白。
- 页面默认控制为一页；内容偏少时密度已舒展，内容偏多时已先做内容取舍而非无限缩小字号。
- 面试材料可在面试前30分钟快速复习。

## 四张覆盖矩阵

内部维护：

1. **JD -> 简历**：能力、来源、对应bullet、证据强度、缺口。
2. **Bullet -> 面试**：讲法、追问、证据口径、风险、覆盖状态。
3. **Claim -> 证据**：类型、来源、置信度、正文/面试处理。
4. **轮次 -> 来源**：依据、置信度、是否生成专属准备。

## 交付判定

- 对用户只交付最终成品文件，不交付 manifest、审计日志、中间稿、渲染图或其他过程文件。
- 文件名使用“姓名-公司-岗位-定向简历.docx”和“公司-岗位-岗位与证据分析.md”等可直接识别的格式；未知字段先询问，不用“final”“最新版”等含糊命名。
- 除非用户另行指定，所有最终成品保存到 Skill 安装目录下的 `output`；脚本应基于自身位置动态解析该目录。

- **Pass**：P0全过，P1全过，P2主要项已处理。
- **Pass with gaps**：P0全过；信息不足的P1项已降级并清楚说明待补内容。
- **Blocked**：任一P0失败，或缺口会导致误导、造假、错市场、漏bullet或臆测轮次。

最终内部回答：

1. 任一claim被追问两层时，用户是否有真实答案或诚实边界？
2. 简历、话术、来源和轮次是否一致？
3. 是否有任何模板残留、内部标签或无法核验的强结论？
